# 第三章 系统需求分析与总体设计

## 3.1 需求分析

### 3.1.1 功能需求

本系统的核心功能是将用户输入的自然语言规范描述转化为标准化的IDS规则文件，并支持对IFC模型进行自动化合规审查。通过对典型海洋工程BIM交付场景的分析，系统需要满足以下功能需求：

**（1）IDS规则生成功能**

系统应支持用户通过自然语言输入规范描述，自动生成符合IDS标准规范的规则文件。主要流程包括：解析用户输入中的实体类型、属性名称、约束条件等关键信息；将解析结果映射至IFC标准实体与属性集；生成符合XML Schema Definition（XSD）规范的IDS JSON结构。该功能是系统的核心能力，需要保证生成结果的准确性与合规性。系统功能模块之间的依赖与调用关系如图3.1所示。

**（2）IFC知识库检索功能**

系统应提供IFC术语知识库的检索能力，支持用户查询特定实体类型、属性集或属性的标准化定义。知识库支持实体名称的模糊匹配与精确查询，支持按IFC版本（IFC2x3、IFC4、IFC4.3）筛选结果，并展示实体的属性集结构、属性类型与约束信息。该功能为IDS生成提供知识支撑，也便于用户验证生成结果的正确性。

**（3）合规审查执行功能**

系统应支持对IFC模型文件执行自动化合规审查。主要流程包括：接收用户上传的IFC文件、调用ifctester加载IDS规则执行审查、生成违规项清单报告，并支持JSON、HTML与BCF三种格式的报告输出。该功能形成"规则生成→审查执行→结果反馈"的完整闭环。

**（4）用户管理与历史记录功能**

系统应提供基础的用户管理能力，包括用户注册、登录验证与会话管理。同时，系统记录用户的IDS生成历史与审查历史，便于用户追溯与复用。历史记录包含输入内容、生成结果、审查报告与时间戳等信息。

### 3.1.2 非功能需求

除核心功能外，系统还需满足以下非功能性要求，确保在实际工程场景中稳定运行：

**（1）性能要求**

系统应在合理时间内完成IDS生成与审查执行。单次IDS生成响应时间应低于10秒（含LLM调用与知识库检索）；对于100MB以内的IFC模型文件，审查时间应低于30秒；系统应支持至少10个并发用户正常访问。

**（2）可靠性要求**

系统应具备基本的容错能力。LLM调用失败时应自动重试或降级处理；知识库检索无结果时应返回明确的提示信息而非异常中断；IFC文件解析失败时应记录错误日志并提示用户检查文件格式。

**（3）安全性要求**

系统应防范常见Web安全风险。用户身份验证采用JWT令牌机制，防止未授权访问；API接口设置速率限制，防止恶意请求耗尽资源；文件上传接口校验文件类型与大小，防止恶意文件注入；敏感操作记录审计日志，便于安全事件追溯。

**（4）可维护性要求**

系统架构应便于后续扩展与维护。前后端分离架构便于独立迭代；核心模块采用组件化设计，便于功能扩展；关键配置参数外置，便于环境迁移；代码结构清晰，便于后续开发者理解与接手。

### 3.1.3 用户角色与使用场景

系统面向三类用户角色：BIM工程师、审查人员与项目管理员。各角色的功能权限详见表3.1，典型使用场景如下：

**BIM工程师**：在模型交付前，通过系统将项目规范转化为IDS规则，验证模型的合规性，及时发现并修正违规项。

**审查人员**：加载项目IDS规则与提交的IFC模型，执行自动化审查，生成合规报告，作为人工复核的辅助工具。

**项目管理员**：管理项目规范库与IDS规则模板，配置审查标准，追溯审查历史记录。

## 3.2 系统架构设计

### 3.2.1 整体架构

系统整体架构如图3.2所示，采用前后端分离的模式，分为三层：前端交互层、后端服务层与数据存储层。

前端交互层基于Next.js框架构建，采用React组件化开发模式，通过API接口与后端通信。该层负责用户界面渲染、输入数据校验、请求发起与结果展示。

后端服务层包含两个子模块：Next.js API Routes与Python FastAPI服务。Next.js API Routes处理轻量级请求（如用户认证、历史记录查询）；Python FastAPI服务处理核心业务逻辑（如IDS生成、知识库检索、审查执行）。两个模块通过HTTP协议通信，形成协同服务体系。

数据存储层采用MongoDB数据库，结合GridFS处理大文件存储。该层存储用户数据、历史记录、IFC知识库与审查报告等持久化信息。

### 3.2.2 前端交互层设计

前端交互层包含以下核心页面与组件：

**（1）登录注册页面**：提供用户身份认证入口，采用JWT令牌机制管理会话状态。

**（2）IDS生成页面**：提供自然语言输入框、IFC版本选择器与生成按钮，展示生成结果与规则预览。

**（3）知识库查询页面**：提供IFC实体搜索入口，展示检索结果与实体详情。

**（4）审查执行页面**：提供IFC文件上传入口与IDS文件选择入口，展示审查报告。

**（5）历史记录页面**：展示用户的生成历史与审查历史，支持结果下载与复用。

前端采用TypeScript语言开发，利用React Hooks机制管理组件状态，通过Axios库发起HTTP请求。界面样式采用Tailwind CSS框架，实现响应式布局。

前端交互层通过API接口与后端服务层通信。考虑到IDS生成涉及LLM调用与知识库检索等计算密集型任务，后端采用Next.js API Routes与Python FastAPI协同的双模块架构，以兼顾开发效率与计算性能。

### 3.2.3 后端服务层设计

后端服务层包含两个子模块，各模块职责如下：

**Next.js API Routes模块**：
- 用户认证接口：处理登录、注册请求，生成与验证JWT令牌
- 历史记录接口：查询与存储用户的生成历史、审查历史
- 配置管理接口：读取系统配置参数（如LLM模型选择、API密钥）

**Python FastAPI模块**：
- IDS生成接口：接收自然语言输入，调用五阶段生成管道
- 知识库检索接口：接收查询请求，执行向量检索与知识图谱匹配
- 审查执行接口：接收IFC文件与IDS文件，调用ifctester执行审查

两个模块之间通过HTTP RESTful API实现通信协作。Next.js API Routes作为前端请求的统一入口，根据请求类型判断：轻量级请求直接处理，核心业务请求则转发至FastAPI模块。前后端交互的完整流程如图3.3所示。

### 3.2.4 数据存储层设计

数据存储结构如图3.4所示，采用MongoDB数据库，其文档型存储模式适合存储结构化但不规则的BIM数据。主要数据集合包括：

**users集合**：存储用户基本信息，包括用户名、密码哈希、创建时间等。

**history_ids集合**：存储IDS生成历史记录，包括输入内容、生成结果JSON、IFC版本、创建时间等。

**history_reviews集合**：存储审查历史记录，包括IFC文件ID、IDS文件ID、审查报告、创建时间等。

**files集合（GridFS）**：存储用户上传的IFC文件与生成的IDS文件。GridFS将大文件分块存储，适合处理数十MB的模型文件。

数据库设计遵循精简原则，仅存储系统运行所需的核心数据，避免冗余信息占用存储空间。

## 3.3 技术选型

各层技术选型及理由详见表3.2，本节对关键技术的选择依据进行阐述。

**Next.js框架**：Next.js是一个基于React的全栈框架，支持服务端渲染（SSR）与静态页面生成（SSG）。选择Next.js的原因在于：其API Routes功能允许在同一项目中实现前后端代码，简化开发流程；SSR特性提升了首屏加载速度与SEO友好性；框架生态成熟，社区支持丰富。

**TypeScript语言**：TypeScript为JavaScript添加静态类型检查，能够在开发阶段发现类型错误，降低运行时异常风险。对于涉及复杂数据结构（如IDS JSON、IFC实体定义）的场景，类型定义提供了清晰的接口约束。

**Tailwind CSS框架**：Tailwind CSS采用原子化CSS类名，通过组合类名快速构建界面样式。相比传统CSS框架，Tailwind避免了样式文件的膨胀，便于实现响应式布局与主题定制。

**Axios库**：Axios是一个流行的HTTP请求库，支持请求拦截、响应拦截与错误处理。相比原生fetch API，Axios提供了更完善的错误处理机制与请求配置能力。

### 3.3.2 后端技术选型

**FastAPI框架**：FastAPI是一个高性能Python Web框架，基于Starlette与Pydantic构建。选择FastAPI的原因在于：其自动生成OpenAPI文档，便于接口调试与文档维护；基于Pydantic的类型验证确保请求参数与响应数据的规范性；异步请求处理能力适合LLM调用等耗时操作。

**LangChain框架**：LangChain是一个LLM应用开发框架，提供提示词模板、链式调用与工具集成等能力。选择LangChain的原因包括：模块化设计便于组装五阶段生成管道；提示词模板管理简化了LLM调用的配置工作；与Facebook AI Similarity Search（FAISS）向量数据库的集成降低了知识库检索的开发成本。

**ifcopenshell库**：ifcopenshell是IFC文件解析的开源库，支持IFC2x3与IFC4版本。该库为知识库构建提供IFC实体定义的提取能力，也为审查执行提供模型数据的访问接口。

**ifctester工具**：ifctester是IDS合规审查的开源工具，能够加载IDS规则并对IFC模型执行自动化检验。该工具作为审查执行引擎，生成结构化的合规报告。

### 3.3.3 数据存储技术选型

**MongoDB数据库**：MongoDB是文档型数据库，其灵活的文档结构适合存储不规则的业务数据。选择MongoDB的原因在于：无需预定义表结构，便于应对数据模型的变化；文档嵌套结构适合存储IDS JSON与审查报告等层次化数据；查询语法简洁，便于实现历史记录的筛选与排序。

**GridFS机制**：GridFS是MongoDB的大文件存储方案，将文件分块存储于两个集合（chunks与files）。选择GridFS的原因在于：MongoDB单文档大小限制为16MB，GridFS突破该限制，适合存储数十MB的IFC文件；分块存储便于文件的流式传输与部分读取。
